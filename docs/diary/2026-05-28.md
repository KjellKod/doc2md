## Mac release CFBundleVersion fix

Context:
- Mac release run 26606697457 failed even after PR #155 merged.
- #155 fixed tag resolution (CFBundleShortVersionString now resolves to 2.6.3 instead of 0.0.0). That success exposed the next failure: the validate gate rejected `CFBundleVersion 1 is reserved for the initial 0.1.0 release`.

Human Inputs to Remember:
- Semver always comes from a manually triggered release in `x.y.z` form, never `vX.Y.Z`.
- We intentionally do not publish the Mac release anywhere yet. Confirmed: the pipeline has never completed a run (appcast latest-download 404s) and the app's `SUFeedURL` is a localhost placeholder with automatic checks disabled.
- Fix it, open a PR ready for review, then run pr-shepherd.

Changes Made:
- `packages/core/scripts/release-version.mjs`: added `deriveBundleVersion(x.y.z)` = major*10000 + minor*100 + patch (2.6.3 -> 20603). Throws on a leading `v` and on minor/patch >= 100. Exposed `bundleVersion` on `getReleaseVersionInfo()`.
- `scripts/build-mac-app.sh`: derive the bundle number and pass `CURRENT_PROJECT_VERSION` to `xcodebuild` alongside `MARKETING_VERSION`. This is the root-cause line: the build never stamped a real build number, leaving the pbxproj default of 1.
- `.github/workflows/release-mac.yml`: replaced the obsolete "1 reserved for 0.1.0" sentinel with a stronger invariant (plist `CFBundleVersion` must equal the value derived from `CFBundleShortVersionString`); fixed the 404 branch to treat a missing appcast as the first publish rather than demanding bundle version 1.
- `packages/core/src/release-version.test.ts`: 6 new tests for `deriveBundleVersion`.

Sources Consulted:
- Failing run log for run 26606697457.
- PR #155 description and the release-mac workflow.
- Published appcast URL (404) and the GitHub release/tag list.

Security Notes:
- No secrets touched. The change does not enable public distribution; Sparkle feed stays a localhost placeholder.

Open Questions:
- The `xcodebuild` plist-stamping step itself can only be verified on a Mac with full Xcode; logic is verified end-to-end otherwise.

Next Step:

## Cramped Safari editor + per-document view memory

Context:
- Branch `doc2md-fix-view`.
- Kjell reported that in hosted Safari the edit/preview panel opens far too short
  vertically and stays cramped, and that switching documents loses both the panel
  size and the caret/scroll position. Asked whether screenshots were needed or the
  issue was findable from code.

Human Inputs to Remember:
- Surface confirmed as hosted web in Safari (not the Mac app).
- Wanted the full fix: reliable auto-size AND per-document memory of size + cursor.

Root Causes (both code-confirmed):
- Auto-size: `recomputeEditShellHeightCeiling` only fired on resize / one rAF / a
  ResizeObserver watching the preview panel, its parent, and `.app-shell`. Entering
  working mode hides the hero and shifts the panel up without changing any observed
  box size, so WebKit could leave the ceiling frozen at the small value measured
  while the hero was still tall.
- Per-doc memory simply did not exist: `editShellHeight` was a single global value
  and caret/scroll were never stored.

Changes Made:
- `useWorkspaceResize.ts`: exposed `recomputeEditShellHeightCeiling`; added
  `documentKey` / `resolveInitialEditShellHeight` / `onEditShellHeightChange`
  options; persist height on drag-end, keyboard, and reset; restore per document on
  `documentKey` change. Moved latest-value ref syncs into an effect (lint:
  no ref access in render).
- `webAdapter.tsx` / `desktopAdapter.tsx`: in-memory per-doc view map (same lifetime
  model as `entrySaveStates`); double-rAF recompute effect on working-mode /
  document change; exposed `getSavedEditorViewState` + `onEditorViewStateChange`.
  Reordered desktop hook init so the selected entry id is available to the resize hook.
- `AppShell.tsx` / `App.tsx` / `DesktopApp.tsx`: threaded the new props through.
- `PreviewPanel.tsx` / `EditMode.tsx`: defined `EditorViewState`; seed a
  `pendingEditorRestoreRef` on entry change (in the entry-reset effect); EditMode
  applies caret + scroll once on mount via `useLayoutEffect` and reports caret/scroll
  back on select/scroll. (Superseded 2026-05-29 by anchor-line restore for both
  preview and edit.)

Decisions Made:
- Per-doc memory is session-only (in-memory map), not persisted across reloads — the
  hosted browser drops loaded files on reload anyway.
- Applied the same wiring to the desktop adapter for parity (Mac app is also WebKit).

Validation:
- `npm ci` (node_modules were absent), `npm run typecheck`, `npm run lint`
  (0 errors; 2 pre-existing desktopAdapter react-refresh warnings), `npm run build`.
- `npm test -- --run`: 720 passed. Added 2 PreviewPanel tests and a new
  `useWorkspaceResize.test.ts`.

Open Questions:
- Manual hosted-Safari confirmation (resolved 2026-05-29: user verified).

Next Step:
- Manual Safari pass, then commit + open/update the PR if it checks out.

Narrative Hook:
- The editor was not actually small; it was measuring itself in the wrong room and
  never looked again once the furniture moved.
