## Quest: PDF Formatting Preservation

**Quest ID:** pdf-formatting_2026-04-06__1833
**Outcome:** Complete (Silver)
**Branch:** claude/suspicious-mestorf

### Context

Kjell wanted doc2md's PDF converter to better preserve formatting from structured PDFs. The converter already had pdfjs-dist positional data available but was not using x-positions, font-size mixing within lines, or multi-page repetition patterns. Seven specific improvements were requested: superscript folding, header/footer stripping, kerning-aware spacing, TOC dot leaders, table detection, nested list indentation, and inline bold spans.

### Human Inputs to Remember

- Kjell explicitly said NOT to commit the test document at `.ws/complicated-pdf` or mention its concrete content in implementation. Use it for local testing only.
- Kjell wants Codex MCP calls to always use `danger-full-access` sandbox to avoid symlink write-boundary issues. Saved as permanent memory.
- Kjell chose full quest workflow, skipped detailed walkthrough, and approved build immediately.

### Changes Made

- Added `foldSuperscripts()` to merge small-font symbols into surrounding text
- Added `detectHeaderFooterItems()` and `stripHeaderFooter()` using real page viewport height with max-y fallback
- Added `isKerningGap()` to suppress false spaces from tight character positioning
- Added `stripDotLeaders()` for TOC cleanup
- Added `groupItemsByRow()`, `getRowColumnClusters()`, `detectTableRegions()`, `renderMarkdownTable()` for table detection via x-position clustering
- Added `computeIndentLevel()` and `isBulletItem()` for nested list indentation with non-body-font bullet detection
- Added `renderInlineBoldText()` for mixed bold/regular spans within lines
- Updated `mergeBulletContinuations()` and `isBodyLine()` to recognize indented bullets
- 21 new tests, 41 total PDF tests, 233 total suite passing

### Sources Consulted

- `docs/implementation/pdf-formatting-analysis.md` for raw positional data evidence and detection rules
- Existing `src/converters/pdf.ts` for current rendering pipeline architecture
- pdfjs-dist TextItem API for available properties (transform, width, fontName, fontSize)

### Security Notes

- No new dependencies added
- No secrets, external I/O, or user input trust boundary changes
- All new code uses only array methods, Math, string operations, and regex

### Open Questions

- Table detection handles simple 2-3 column tables. Complex tables with merged cells and multi-line cells are deferred to a future phase.
- The ~18pt indent step for nested lists is a constant. May need per-document auto-detection for PDFs with different margin structures.

### Narrative Hook

Table detection died twice. First it treated every distinct x-position as its own column, which meant kerned text or bold labels inside cells would blow up the column count. Then it came back with proper clustering but forgot that `hasEOL` was the original line boundary signal, nearly collapsing same-y lines. The third version finally got both right. Dexter called it "a table detector still carrying bad instincts" in the requiem, which is fair. The instincts just needed two surgeries.
