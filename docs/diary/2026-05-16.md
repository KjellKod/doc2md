# Diary: 2026-05-16

## paste-to-markdown order preservation (PR #130 follow-up)

Context:
- Branch `quest/paste-to-markdown`, worktree `.worktrees/quest/paste-to-markdown`.
- User reported that copying rich Google Docs content into doc2md and then copying the resulting text changes visible order: top-level items like "At least 65%", "Security Automatic Pipeline", "Create a timeline", "Raising CI score", and "Month of June" appeared before ONF child items that were earlier in source. Children were dumped as a block after all parents.
- Earlier commits on this branch flattened weak levels and capped nesting at one level, but ordering was still wrong on real paste because the DOM reconstruction passes reorganized nodes.

Human Inputs to Remember:
- Source order is the highest invariant. A correctly ordered single-level list beats a nested-but-reordered list every time.
- Weak hints like `lst-kix_*`, `li-bullet-*`, and copied CSS `margin-left` must not be allowed to create arbitrary or order-breaking structure.
- Scope is strictly editor paste conversion. Copy and export paths must not be touched.

Changes Made:
- Replaced `nestGoogleDocsLists()` and `nestGoogleDocsFlatListItems()` (both moved DOM nodes) with `annotateGoogleDocsListLevels()` (annotation-only). Each `<li>` gets a `data-doc2md-list-level` attribute from the parent `<ul class="lst-kix_*-N">` first, then `li-bullet-N`, then `aria-level`, clamped by `maxGoogleDocsInferredListDepth`.
- Added a `googleDocsListItem` Turndown rule that emits the appropriate bullet or ordered prefix and indents content by `level * 4` spaces. Source DOM order is preserved verbatim because nothing moves.
- Added `collapseBlankLinesBetweenListItems()` post-processor so consecutive Google Docs sibling lists render as one continuous list, not two list blocks separated by a blank line.
- `pasteHtmlNormalizer.ts` now also prefers the parent `lst-kix_*-N` class when deciding the list-item level, matching the converter logic.
- Regression tests cover four cases: batched-by-level sibling lists (verbatim DOM order), interleaved level-0/level-1 sibling lists (the reported bug pattern, ONF-9505 before "At least 65%", Security after DEV-2936), `0,1,0` pattern, and `0,1,2,1,0,1,1,1` pattern with depth cap.
- Single commit on the branch: `14dea0e Preserve Google Docs paste order via annotation, not DOM moves`.

Open Questions:
- The "batched-by-level" test case documents that when Google Docs emits all level-0 items in one `<ul>` followed by all level-1 items in another, we keep verbatim DOM order rather than try to reattach children to the right parents. That mirrors the source order but does not reproduce the original document's visual nesting. The real-world Google Docs HTML observed so far is the interleaved pattern, which is the case the bug report cared about. Worth re-verifying against fresh Google Docs HTML samples if the user reports another regression.
- PR #130 still wants a manual paste in the running app to confirm the editor renders the new output correctly. Targeted unit and e2e coverage is in place but a human eyeball pass on the actual document the user copied would close the loop.

Next Step:
- Push the commit, let the existing PR pick it up, and ask the user to paste the original Google Docs source into a fresh scratch entry to confirm the editor preserves the order they expect.

Narrative Hook:
- The fix sentence is short: stop moving things. Annotate the level, let the Markdown emitter do the rest. The previous attempts traded correctness for cleverness, and the cleverness kept rewriting the user's source order. Annotation is dull, and dull is exactly what a paste pipeline should be.
