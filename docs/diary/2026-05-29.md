## Safari PDF conversion broke after the pdfjs 5 bump (#132)

### Context
- After #132 (pdfjs 4.10.38 -> 5.7.284), Kjell saw PDFs convert fine on the
  deployed site and via the CLI, but fail on localhost with "We couldn't read
  this file" (red "Failed" / "Poor"). A pre-#132 build worked locally.
- First instinct (mine) was a stale Vite/node_modules cache producing an
  API/worker version mismatch. Kjell pushed back: clean `npm ci` on the branch,
  still failing. My fresh headless repro (Chromium AND Playwright WebKit, dev
  AND production preview) all PASSED — so the code was fine and I was chasing
  the wrong ghost.
- The real split surfaced from one question: works in Brave (Chromium), broken
  in Safari (WebKit), fresh server + hard reload. Safari 26.4, bleeding edge.

### Root cause
- pdfjs 5's `page.getTextContent()` (pdf.mjs:22118) consumes its text stream
  with `for await (const value of readableStream)` over a NATIVE ReadableStream.
  WebKit is the only engine that does not implement
  `ReadableStream.prototype[Symbol.asyncIterator]` (webkit.org #194379), so that
  line throws "undefined is not a function (near '...value of readableStream...')".
  pdfjs 4 did not stream text this way -> Safari-only regression. Playwright's
  bundled WebKit HAS the iterator, which is why every automated surface lied.
- Confirmed from the actual Safari Web Inspector stack (surfaced only after I
  added a `console.error` to convertPdf's bare `catch {}`, which had been
  swallowing the real error and impersonating a corrupt file).

### Changes Made
- `src/converters/pdf.ts`: `ensureReadableStreamAsyncIterator()` — feature-
  detected, idempotent polyfill installing the spec-shaped async iterator only
  when missing; wired into `ensureBrowserWorkerConfigured()` before getDocument.
  Plus `console.error(...)` in the convertPdf catch.
- `src/converters/pdf.test.ts`: 2 regression tests (installs-when-missing,
  idempotent/no-clobber).
- `src/styles/global.css`: collapsed the per-format badge palette — all
  supported types now green (PDF/XLSX no longer borrow error-red), unknown ->
  neutral grey. Net -80 lines. The status icon carries success/failure now.
- Two commits (Safari fix; badge recolor). PR #162, CI fully green
  (lint/type, test, build, e2e), marked ready for review.

### Human Inputs to Remember
- Kjell pushes back with evidence and is right to: "I ran npm ci, still fails."
  Don't anchor on the first plausible theory — reproduce the actual failure
  before prescribing. The browser split (Brave vs Safari) was the whole key.
- Kjell verified the fix in real Safari himself ("works now in safari too").
- Delegated the badge color choice to me; wanted supported docs to not look
  like errors. Green-for-all-supported was the right read.

### Sources Consulted
- node_modules/pdfjs-dist/legacy/build/pdf.mjs (getTextContent / streamTextContent)
- node_modules/pdfjs-dist/build/pdf.worker.mjs (apiVersion/workerVersion guard)
- Safari Web Inspector console stack (provided by Kjell)
- webkit.org bug #194379 (ReadableStream async iteration unimplemented)

### Security Notes
- Polyfill is feature-detected and spec-shaped; no global behavior change on

### Open Questions

## Follow-up: unify cross-document position restore (preview + edit)

Context:
- Continuation of the 2026-05-28 cramped-Safari + per-document-memory work
  (branch `doc2md-fix-view`, commit `53bcb55`), before the PR was pushed.
- Kjell verified the editor auto-size and the edit-mode caret restore in Safari,
  then flagged a real gap: returning to a document opens in **preview** and the
  preview was stuck at the top of the document. Only edit mode restored position.

Human Inputs to Remember:
- Position preservation across edit/preview (same doc) and across document
  switches is CORE behavior — anything that jolts you to the top is unacceptable.
- Explicitly asked for e2e coverage of this behavior, and was surprised it could
  regress without an e2e failure.

Diagnosis:
- The first cut only restored the edit textarea caret/scroll. Preview uses a
  different, mode-agnostic currency — the `pendingAnchorLineRef` source line that
  `switchMode` hands off and `useRenderedAnchorApply` consumes. Cross-document
  return never seeded it, so preview opened at line 1.
- Same-document edit<->preview was NOT broken (switchMode seeds the ref in an
  event handler before the new mode mounts); the existing
  `view-anchor-mode-switch.spec.ts` still passed. The cross-document case simply
  had no coverage because the feature was new.

Changes Made:
- `EditorViewState` now carries `anchorLine` (top source line) plus the optional
  exact edit caret/scroll. PreviewPanel captures the anchor line in whatever mode
  is active (added `onReportView` + `onScroll` to PreviewMode/LinkedInMode; edit
  reports on scroll/select) and persists per document.
- On document change PreviewPanel seeds `pendingAnchorLineRef` (and the edit
  caret) DURING render via a derived-ref handoff — necessary because parent
  effects run after child mode effects, and the common case (preview stays
  mounted across a switch) consumes the ref on the same commit. Mirrors how
  switchMode seeds synchronously.
- Result: returning to a document restores position in preview AND edit; the
  exact edit caret is still restored when the doc was left in edit mode; same-doc
  switching is untouched.

Tests:
- New e2e in `tests/e2e/view-anchor-mode-switch.spec.ts` (describe
  "cross-document view anchor"): preview scroll restored on return (with an
  explicit `restoredTop > 1` regression guard), and editor caret restored on
  return. All 10 anchor e2e pass on chromium; the 8 pre-existing same-doc tests
  confirm no regression.

Validation:
- `npm run typecheck`, `npm run lint` (0 errors; 2 pre-existing desktopAdapter
  react-refresh warnings), `npm run build`.
- `npm test -- --run`: 720 pass (deterministic after warmup). Cold first-run
  flakes seen in unrelated timing tests (`noticeInventoryDrift`, `InstallPage`,
  PreviewPanel copy) — each passes on rerun; not caused by these changes.
- `npx playwright test --project=chromium tests/e2e/view-anchor-mode-switch.spec.ts`: 10 passed.

Open Questions:
- None functional. Worth a follow-up someday: the cold-start unit-test flakiness
  in child-process/timer tests under parallel load.
