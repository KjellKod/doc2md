## Mac CI signing: "Unknown format in import"

### Context
- The Mac release signing job has never succeeded in CI. The 2.6.x version/tag
  fixes (#155, #156) finally let a run reach the signing step (2.6.4), which
  then died at `security import` with
  `security: SecKeychainItemImport: Unknown format in import.`
- Kjell believed signing "works locally." It does, but that path never proves
  the CI path: `release_mac_local.sh` signs with an identity already in the
  login Keychain and never base64-decodes anything. CI is the only path that
  decodes the `MACOS_CERTIFICATE` secret and runs `security import`.

### Human Inputs to Remember
- Analyze first, do not guess, do not change anything before agreement. Honored:

- Compare with the sibling repo `../candid_talent_edge`; same secret/p12, and
  "no difference how we build locally vs CI." That comparison was the key: the
  sibling strips `\n\r ` before decoding and documents that BSD base64 on the
  macOS runners mishandles embedded newlines. doc2md did not strip.
- Make the robust fix, run all tests, self-review + Codex review, address
  valuable findings, open a PR ready for review, then run pr-shepherd. Full
  permissions granted.

### Changes Made (PR #160, branch fix/mac-ci-p12-base64-decode)
- `scripts/release/sign_mac_app.sh`: strip CR/LF/spaces before
  `base64 --decode`; fail loud at the decode site if the result is empty or its
  first byte is not `0x30` (the ASN.1 SEQUENCE tag every DER PKCS#12 begins
  with). Moved the `uuidgen` keychain-password default into `create_keychain`
  so sourcing the script has no side effects.
- `scripts/release/notarize_mac_app.sh`: same `tr -d '\n\r '` on the base64
  `.p8` branch; raw `BEGIN PRIVATE KEY` passthrough unchanged.
- Added a source-guard so the signing functions can be unit-tested without
  running the signing flow.
- `tests/release/test_p12_decode.sh`: regression guard (wrapped/noisy base64
  round-trips, empty + non-p12 rejection).
- `.github/workflows/ci.yml`: run the release script tests on PRs.
