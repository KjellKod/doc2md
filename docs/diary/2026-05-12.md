## Security Notes

- The threat model adopted is single-maintainer, trusted-vendor (OpenAI). Reviewer gates and SHA pinning of `actions/*` were deliberately rejected because they trade real maintenance overhead against threats that are either non-existent (single maintainer) or platform-level (SHA pinning doesn't help against GitHub-platform compromise).
- The Codex CLI install in `codex-ci-review.yml` remains unpinned by maintainer preference. The `--ignore-scripts` flag closes the lifecycle-script half of the attack surface. Residual risk: a compromised `@openai/codex` minor release could execute arbitrary JS once `codex exec` runs against a prompt. The blast radius then is "what codex's sandbox allows" — read-only, no GITHUB_TOKEN in env at exec time — not "anything the runner can do during install." This is a reasonable trade for latest-always.
- The new `codex-version-drift.yml` workflow is its own potential supply-chain surface but is locked down: no `npm install` of any kind (uses `npm view` only, which queries the registry without executing package code), `if: github.repository == 'KjellKod/doc2md'` to prevent forks from opening upstream issues, `permissions: contents: read, issues: write` minimum.
- Accepted trade-off: if the maintainer manually closes the drift tracking issue without updating `.github/codex-cli-version.txt`, the next drift run opens a new issue rather than reopening the closed one. Reopening a closed issue would override deliberate maintainer dismissal.

## Open Questions

- Should we add `npm audit` to one of the workflows as a periodic check, separate from Dependabot? Dependabot opens PRs on known CVE alerts but doesn't run audit interactively. Out of scope for this quest; worth a follow-up think.
- The new guard test suite (15 rule-prefixed tests + 9 prior tests = 24 total) needs maintenance discipline going forward. Worth asking: when a future workflow adds a legitimately new pattern (e.g., a new release-step that needs `cache: npm`), the guard rule false-positives by design. The accepted answer is to widen the allowlist in `security_ci_guard.py`, not to weaken the rule predicate. Documented in `.github/codex-cli-version.txt` adjacency? Maybe a `docs/security/guard-allowlist-policy.md` is overdue.
