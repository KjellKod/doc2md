# Diary: 2026-05-20

## External link routing in web + desktop (PR #138)

Context:
- Branch `links`. User report: clicking links in the desktop app does nothing or replaces the app shell. Three classes of failing links in the wild: anchors inside converted documents (rendered by react-markdown), Install page download buttons (`.tgz`, `.skill`), and About-section chrome links (GitHub, sponsor, etc.).
- Root cause was identical in all three cases: the Mac shell loads the SPA inside a `WKWebView` pinned to a custom `doc2md://` scheme, with no `WKNavigationDelegate.decidePolicyFor` and no `WKUIDelegate.createWebViewWith`. Regular anchors replaced the SPA; `target=_blank` silently no-opped; `<a download>` had nothing to deliver because the `.tgz` and `.skill` files are never bundled into the Mac app's `Resources/Web` directory.
- Web app had its own latent bug: clicking a markdown body link replaced the SPA and lost the user's working state.

Human Inputs to Remember:
- Design principle the user picked, in their own words: "I want to avoid having our desktop app trying to be a web browser for all sorts of things." Translation that drove the implementation: every external link opens *outside the current view* — a new browser tab on web, the system default browser on macOS. The desktop is a thick client around the converter, never a general browser. Same rule everywhere; no web-vs-desktop branching at the link level.
- Workflow ask: "just do it, then review it and address any findings, use /codex:review also, drive our changes with quality, once finished internally and reviewing then create a PR with /pr-assistant and monitor and fix things with /pr-shepherd until it's completed." Auto mode active throughout.

Changes Made:
- `apps/macos/doc2md/WebShellLinkPolicy.swift` (new) — pure URL-classification enum exposing `isInternalURL`, `isExternallyOpenable`, `route(for:)` returning a `Routing` decision, and `route(forNavigationActionWith:navigationType:)` that gates external opening on `.linkActivated` only. Allows `doc2md://app/...` (host pinned), and in DEBUG additionally `http://localhost:5173/` (port pinned, host case-insensitive).
- `apps/macos/doc2md/WebShellView.swift` — `Coordinator` now `WKUIDelegate`, holds an injectable `ExternalURLOpener` (default `NSWorkspace.shared.open`), and routes both `decidePolicyFor navigationAction` and `createWebViewWith` through `WebShellLinkPolicy.route(...)`. Both delegate methods are 3-line wrappers.
- `apps/macos/doc2mdTests/WebShellLinkPolicyTests.swift` (new) — 36 tests covering scheme/host classification, route decisions, navigation-type filtering, DEBUG localhost rule.
- `apps/macos/doc2md.xcodeproj/project.pbxproj` — added the two new Swift files to both the main and test source build phases.
- `src/components/preview/PreviewMode.tsx` — `components.a` override on ReactMarkdown sets `target="_blank" rel="noopener noreferrer"` on every rendered markdown anchor. Mdast `node` prop is destructured out so it never reaches the DOM.
- `src/components/__tests__/markdownExternalLinks.test.tsx` (new) — regression coverage for plain links, mailto, relative, links wrapping inline code, links wrapping images, and a no-leaked-`node`-prop assertion.
- `src/components/InstallPage.tsx` — when `useDesktopCapability().isDesktop` is true: skip the manifest fetch, hide the `.tgz` and `.skill` download buttons, replace them with a single external link to `https://kjellkod.github.io/doc2md/` and a one-sentence explanation.
- `src/components/InstallPage.test.tsx` — two new tests asserting the desktop variant skips the fetch and shows the right link with `target=_blank rel="noopener noreferrer"`.
- `.quest-manifest` — added `.agents/skills/gpt/SKILL.md` (preexisting branch blocker for `scripts/validate-manifest.sh`).

Security Notes:
- The policy layer narrows blast radius in two places `AppSchemeHandler` does not: it pins the `doc2md://` host to `app`, so `doc2md://attacker/index.html` is no longer treated as in-shell, and it pins the DEBUG localhost rule to port 5173 so a markdown link to `http://localhost:3000/` cannot turn the shell into a local browser. External handoff is limited to `http`, `https`, `mailto`, `tel`; `file:`, `javascript:`, `about:` are canceled silently. Only `.linkActivated` navigation actions externally open — form submissions, back/forward, reloads, and `.other` programmatic navigations are canceled silently so a POST body cannot be replayed as a Safari GET.

## Merge Update (same day)

PR #138 merged as squash commit `cd96916` on `main`. Final shape exceeded the original brief because three follow-ups landed on the same branch during review and were addressed inline rather than punted to fresh PRs:

- **rehype-slug + 3-bucket link classifier.** Internal hash anchors (`#mac-desktop-app`) were rendering as `<a href="#…">` without a matching heading id, so clicks scrolled to the top of the page. Added `rehype-slug` to the rehype plugin chain (placed before `sourceLineRehype` and `findHighlight`) so every heading gets a github-slugger id. Repo-relative paths (`../apps/macos/README.md`) were navigating the webview to a 404 and producing a blank window. Refactored `externalMarkdownHref` into a three-bucket `classifyMarkdownHref` returning `external | anchor | disabled`. Disabled renders as a visibly-muted link, dotted underline, `cursor: not-allowed`, `aria-disabled="true"`, `tabIndex=-1`, `onClick`/`onAuxClick`/`onKeyDown(Enter, Space)` all `preventDefault`. Original `href` preserved so right-click → Copy Link still works against the source repo on GitHub.
- **CSS-only tooltip + scroll-margin-top.** User flagged that native HTML `title` attribute tooltips are slow and OS-styled; saved that as a durable memory (`feedback_no_native_tooltips.md`). Swapped the disabled-link `title=` for the project's existing sibling-`<span role="tooltip">` pattern (used by `StatusIndicator`, `FileListItem`, `PreviewToolbar`). Also added `scroll-margin-top: 88px` to `h1–h6` under `.markdown-surface` so hash-jumps land below the sticky preview toolbar instead of behind it.
- **Desktop large-paste auto-shrink regression.** User noticed pasting a large doc into a fresh draft no longer collapsed the hero. Traced to PR #135's AppShell refactor: the desktop adapter got an `onLargeMarkdownPaste: () => {}` stub. Mirrored the web adapter's `pastePromotedEntryIds` pattern in `desktopAdapter` and wired `handleLargeMarkdownPaste`. Followed strict TDD per the user's ask: failing desktop vitest test first (the `is-working-mode` class never appeared), then the fix, then re-ran without changing the test and watched it pass. Also added a Playwright spec (`tests/e2e/large-paste-auto-shrink.spec.ts`) covering the same flow on the web build with both positive and negative cases so future regressions are caught on either shell.
