# Diary: 2026-05-20

## External link routing in web + desktop (PR #138)

Context:
- Branch `links`. User report: clicking links in the desktop app does nothing or replaces the app shell. Three classes of failing links in the wild: anchors inside converted documents (rendered by react-markdown), Install page download buttons (`.tgz`, `.skill`), and About-section chrome links (GitHub, sponsor, etc.).
- Root cause was identical in all three cases: the Mac shell loads the SPA inside a `WKWebView` pinned to a custom `doc2md://` scheme, with no `WKNavigationDelegate.decidePolicyFor` and no `WKUIDelegate.createWebViewWith`. Regular anchors replaced the SPA; `target=_blank` silently no-opped; `<a download>` had nothing to deliver because the `.tgz` and `.skill` files are never bundled into the Mac app's `Resources/Web` directory.
- Web app had its own latent bug: clicking a markdown body link replaced the SPA and lost the user's working state.

Human Inputs to Remember:
- Design principle the user picked, in their own words: "I want to avoid having our desktop app trying to be a web browser for all sorts of things." Translation that drove the implementation: every external link opens *outside the current view* — a new browser tab on web, the system default browser on macOS. The desktop is a thick client around the converter, never a general browser. Same rule everywhere; no web-vs-desktop branching at the link level.
- Workflow ask: "just do it, then review it and address any findings, use /codex:review also, drive our changes with quality, once finished internally and reviewing then create a PR with /pr-assistant and monitor and fix things with /pr-shepherd until it's completed." Auto mode active throughout.

Allows `doc2md://app/...` (host pinned), and in DEBUG additionally `http://localhost:5173/` (port pinned, host case-insensitive).
- `apps/macos/doc2md/WebShellView.swift` — `Coordinator` now `WKUIDelegate`, holds an injectable `ExternalURLOpener` (default `NSWorkspace.shared.open`), and routes both `decidePolicyFor navigationAction` and `createWebViewWith` through `WebShellLinkPolicy.route(...)`. Both delegate methods are 3-line wrappers.

Security Notes:
- The policy layer narrows blast radius in two places `AppSchemeHandler` does not: it pins the `doc2md://` host to `app`, so `doc2md://attacker/index.html` is no longer treated as in-shell, and it pins the DEBUG localhost rule to port 5173 so a markdown link to `http://localhost:3000/` cannot turn the shell into a local browser. External handoff is limited to `http`, `https`, `mailto`, `tel`; `file:`, `javascript:`, `about:` are canceled silently. Only `.linkActivated` navigation actions externally open — form submissions, back/forward, reloads, and `.other` programmatic navigations are canceled silently so a POST body cannot be replayed as a Safari GET.

Narrative Hook:
- The whole branch turned on one principle the user named in a single sentence: the desktop is a thick client, never a browser. Everything else — the navigation-type filter, the host pin, the port pin, the createWebViewWith handoff, the markdown anchor override, the InstallPage swap — is just that sentence expressed in code in the right places. The fun part was that Codex caught me trying to be clever twice: once by dropping `node` from the destructure (which silently leaked it to the DOM), and once by being permissive in DEBUG about localhost ports (which silently turned the shell into a local browser). Both were "minor" cleanups that quietly violated the principle. The principle won.
