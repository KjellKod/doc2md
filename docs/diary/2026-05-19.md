# Diary: 2026-05-19

## Trivial-HTML-wrapper paste detection

Context:
- Branch `markdown-copy`.
- User pasted markdown text from a webpage into Gmail, used "remove formatting", then copied from Gmail into doc2md. Gmail's clipboard still emits HTML (each visible line wrapped in a `<div>`), so the editor's paste handler routed it through `convertHtmlFragmentToMarkdown` (Turndown).
- Turndown's default behavior is to escape markdown punctuation in text nodes so plain-text HTML cannot accidentally produce formatting. That's correct for genuine rich text but catastrophic when the HTML's text is itself markdown: every `#`, `**`, `` ` ``, `[`, `]`, `_` got escaped (`\#`, `\*\*`, `\``, `\[`, etc.) and every `<div>` became a trailing two-space hard break, producing the garbage output the user reported.
- User confirmed that pasting with Cmd+Shift+V (which makes the browser drop the HTML payload and only deliver `text/plain`) already produced correct markdown. That validated the diagnosis: the plain-text path is correct; the HTML path is the regression.

Human Inputs to Remember:
- Don't make changes until analysis and a clear suggestion are agreed.
- Prefer silent smart detection over a pre-paste modal: a "paste mode" picker would punish the 95% common case (rich text from a web page) to handle the 5% case.
- The trailing two-space hard breaks ("    ") that Gmail injects on every line are a related but separate concern; out of scope for this fix.

Changes Made:
- Added `htmlIsTrivialPasteWrapper(html, plainText)` in `src/components/preview/pasteToMarkdown.ts`. Returns true when the HTML body contains none of a fixed set of structural/semantic tags (`h1-h6`, `strong`, `b`, `em`, `i`, `u`, `s`, `code`, `pre`, `blockquote`, `ul`, `ol`, `li`, `table`/`tr`/`td`/`th`, `img`, `hr`, `a[href]`, etc.) and no inline `style` declarations encoding bold/italic, AND the HTML's text content equals the plain-text payload after whitespace reduction. Comparison is whitespace-collapsed character bag, because `textContent` does not insert whitespace at block boundaries while the plain-text clipboard payload does.
- Wired into `convertClipboardPasteToMarkdown`: when HTML is present but is a trivial wrapper, fall through to the plain-text path (`convertLinkedInUnicodeToMarkdown(plainText)`).
- Tests:
  - New: Gmail-style `<div>line</div>` wrapping of the reported battery-alert markdown payload (fenced code, headings, bullets, ordered list, bold) — routes to plainText, output is verbatim.
  - New: Span-wrapped lines with decorative `font-family`/`color` styles (not bold/italic) still classified as trivial wrapper.
  - New: `<strong>` in the HTML disables the wrapper path and keeps the HTML conversion path.
  - New: HTML and plain text disagreeing on characters (Google-Docs-style em-dash auto-substitution) keeps the HTML conversion path.
  - Updated four existing tests that previously asserted the HTML escape pipeline ran on trivial wrappers — they now correctly assert `source: "plainText"` and verbatim passthrough. The escape-`---` → `\---` behavior only fires now when the HTML and plain text disagree (the real Google Docs auto-substitution case), which is the more honest semantics.

## Follow-up: Gmail-signature blind spot

Context:
- User manually verified the first fix worked. Reported one remaining failure mode: when the Gmail signature (profile image plus styled "Kjell Hedstrom" name) was included in the copied selection, the wrapper heuristic disqualified the entire paste and the HTML path mangled everything again. The signature's `<img>` and `<b>` tripped the "has meaningful formatting" check.
- Screenshot confirmed signature structure: an `<img>` and bold colored span. Visible textContent ("Kjell Hedstrom") is present in plain text too, so the character-bag comparison still matched — only the formatting check disqualified.

Changes Made:
- Added `plainTextHasStrongMarkdownSignals(plainText)` in `pasteToMarkdown.ts`. Counts hits across nine distinct markdown patterns (ATX heading, fenced code, link, bold via `**`/`__`, inline code, two-or-more bullet items in a row, two-or-more ordered items in a row, hr, blockquote). Returns true on ≥ 2 distinct hits.
- Restructured `htmlIsTrivialPasteWrapper` so that the character-bag match is the gate, then either (a) no meaningful formatting OR (b) plain text has strong markdown signals triggers the wrapper path. Defense-in-depth pattern: extraneous HTML decoration (signature, mail tracker, copy footer) no longer overrides clearly-markdown plain text.
- Two new tests: one with the user's exact failure mode (markdown body + Gmail signature `<img>` + styled name → routes to plainText, verbatim output, no escapes). One regression guard: a single markdown-looking line in plain text is NOT enough to override real rich HTML (heading + bold + link still goes through HTML path).
