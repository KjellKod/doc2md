# 2026-05-10 — Post-2.3.1 release follow-up (PR #113)

## Context

Tag `2.3.1` was pushed right after PR #112 merged. KjellKod reviewed the resulting Actions runs and flagged three things:

1. Confirm DMG builds never trigger on a plain PR merge, only on a canonical `X.Y.Z` tag push (no `v` prefix).
2. The workflow run title showed the merge commit subject ("Add professional DMG packaging and release validation"). It should show something like `Release/Deployment: <X.Y.Z>`.
3. `Mac release` run `25639154825` failed with `startup_failure` because `actions/download-artifact@<sha>` was not in the repo allowed-actions list.

Also surfaced mid-PR: `release_mac_local.sh` was emitting `doc2md-$(MARKETING_VERSION).dmg` because it read the source Info.plist (which holds Xcode placeholders) instead of resolved version.

## Changes Made

- `.github/workflows/release-mac.yml`: added `run-name: "Release/Deployment: ${{ inputs.tag || github.ref_name }}"`; unpinned SHAs to tag form in `build` and `publish` jobs; kept SHA pins on `package-mac` and `sign-sparkle` (release-secret-bearing) jobs.
- `.github/workflows/deploy-pages.yml`: added `run-name: "Release/Deployment: ${{ inputs.ref || github.ref_name }}"`.
- `.github/workflows/mac-pr-check.yml`: unpinned SHAs to `@v5` / `@v6` / `@v7` (PR check job, no release secrets).
- `.github/workflows/ci.yml`: bumped stale `actions/upload-artifact@v4` to `@v7` (Playwright failure-artifact step).
- `scripts/release/release_mac_local.sh`: replaced the source-plist VERSION derivation with a git-based one. Working tree clean AND HEAD on the latest `X.Y.Z` tag uses the tag verbatim; otherwise bump patch and append `-dev`.

## Security Notes

- `python3 scripts/security_ci_guard.py` passed after the workflow edits.
- Defense-in-depth preserved: `package-mac` and `sign-sparkle` still SHA-pin both `actions/checkout` and `actions/{upload,download}-artifact`. Tag-mutation in `actions/checkout@v5` cannot reach the Apple cert or the Sparkle EdDSA key.
- Allowlist requires three SHA pins (`checkout`, `upload-artifact`, `download-artifact`) for the secret-bearing jobs; called out as `[!IMPORTANT]` in the PR body so KjellKod adds them in the GitHub UI before the next tag push.
- `release_mac_local.sh` does not auto-fetch tags; a stale local clone fails closed with a clear message.

## Open Questions

- Should `package-mac` job also gain coverage in `security_ci_guard.py`? The current regex only matches `secrets\.APPLE_[A-Z0-9_]+` and `secrets\.SPARKLE_[A-Z0-9_]+`, so it does not flag the `MACOS_*` / `NOTARIZE_*` secret names actually used in `package-mac`. The job is SHA-pinned today by convention, but the guard would not catch a regression. Worth a small follow-up.
- Whether `release_mac_local.sh` should bake the resolved `VERSION` into the built app via `xcodebuild MARKETING_VERSION=…` so the app's internal version matches the filename. Currently the app keeps Xcode's `MARKETING_VERSION` and only the filename gets the `-dev` suffix. Acceptable for unpublished local builds; flag if user wants internal consistency too.

## Narrative Hook

KjellKod's pinning policy was clean and reasoned — "trust official actions on tag form" — and ninety-nine percent of the workflows aligned cleanly. The interesting moment was the security guard biting back on `sign-sparkle`: a deliberate stricter rule sits one layer underneath the general policy, specifically guarding the path that holds the Sparkle private key. The two rules don't conflict, they compose. The general policy says "trust tag form for official actions"; the guard says "except in this one tiny corner where the blast radius is the entire updater channel". I almost preached the SHA-pin tradeoff in the abstract, then the codebase made the argument for me with a `[FAIL]` line. Tail line: defense-in-depth speaks softly and carries a regex.
