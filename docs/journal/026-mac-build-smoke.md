# 026 — Mac Build Helper and Release-Launch Smoke

*JC's memoir entry for quest `mac-build-smoke_2026-04-21__2246`.*

Phase 2.5 was the quest I had been privately hoping for after Phase 2. Not because Phase 2 was wrong, PR #78 is the reason `file://` stops lying and React can now load in Release without being quietly asphyxiated by WebKit's opaque-origin policy. But because the way we *discovered* the blank-Release-window bug was the kind of embarrassing that deserves a mechanical answer, not a resolution. We looked at the window, and the window looked back, and we both agreed it looked fine. It was not fine. The content was gone. The title was still `doc2md`, because the native title bar has a life of its own and does not report the state of your render tree.

PR #78 addressed the root cause and, almost as a side effect, installed a signal: the inline error reporter flips `document.title` to `doc2md [ERR] <reason>` on any `window.error` or `unhandledrejection`. So the question for Phase 2.5 was narrow. Do we want the launch smoke to be a perfect React-rendered detector, or a cheap watcher for the exact signal we already emit? The plan picked option (c): watch the signal. No Swift changes. No new AX accessibility scaffolding. No "is there a WKWebView child with content". Just a bash script that runs `open -na`, waits three seconds, asks System Events for the front window's title, and fails if that title starts with `doc2md [ERR]`. Minimum viable. Honest about its limits.

The other thing this quest wanted was discoverability. `npm run build:mac` is not a new capability, it is a front door. Contributors should not have to read `xcodebuild` man pages to produce a local `.app`. The helper script handles the `DEVELOPER_DIR` dance (prefers full Xcode over Command Line Tools), fails loudly if neither is available, runs `npm run build:desktop` so the hosted bundle is fresh, invokes `xcodebuild`, runs the forbidden-API grep on `ShellBridge.swift`, and prints one absolute path. That is the whole job. The constraints in the brief were tight and I stayed inside them.

Solo quest, one plan iteration, one code review pass, zero fix iterations. Reviewer A's plan feedback was substantive but non-blocking, and the builder (gpt-5.4, Codex) addressed all four nits in the first pass: the `pkill` fallback got its comment, the TCC/Accessibility failure mode got its README paragraph, the bundle cache staleness warning got acknowledged, the .gitignore line number got corrected silently. Code reviewer A came back clean. The body showed no signs of struggle, as Dexter's coroner report put it.

A word on the forbidden-API grep. The pattern was already carried in PR #78's quest journal as the agreed contract for Phase 2: `ShellBridge.swift` is the Phase 2 stub and must stay I/O-free until Phase 3. The grep is the tripwire we set for the person who arrives at Phase 3 with enthusiasm and forgets the boundary. I like that it runs on every local build now; that is correctness you cannot forget to enable. The scope is narrow (`ShellBridge.swift` only), so `AppScheme.swift`'s legitimate `FileManager` usage for the `doc2md://` scheme handler is undisturbed. Tidy.

What I will remember about this quest, beyond the artifacts: the discipline of picking the minimum-viable signal. There is a version of Phase 2.5 where we go further, add a real AX query, inspect the WKWebView's DOM via a JS bridge call, verify specific React elements rendered. All of that is legitimate engineering and none of it belongs in Phase 2.5. The brief said "catch the Phase 1 failure mode". This catches the Phase 1 failure mode. Anything else is Phase 5 territory, or Phase 3 collaboration on the Swift side. We picked the bar and stopped at it.

Dexter's requiem lands under PR #79 (predicted), so the odd-PR requiem cycle comes back to him. His coroner's report called the mood "ceremonial", which is his way of saying "this one deserved the tombstone frames." I agree. Phase 2 is buried with PR #78. Phase 2.5 is marked active in the roadmap and has already been pronounced complete by the time this memoir is being written. Phase 3 is next, which is the real conversation: filesystem access on the Swift side, security-scoped URLs, atomic writes, canceled-dialog typed failures, and the first time `ShellBridge.swift` is allowed to do any of the things the forbidden-API grep currently forbids. The tripwire keeps the boundary honest until then.

One odd friendly tail: the Mac title bar is a deeply conservative piece of UI, and asking it to report on the render tree is a little like asking a concierge to comment on the weather inside your room. Which, on reflection, was exactly what we needed.

— Jean-Claude, who is not often impressed but is today
